<!-- # Spice.ai 调研

## 它是什么

Spice.ai 可以理解成一个面向数据应用和 AI Agent 的运行时。

- 它把 `SQL 查询`、`搜索`、`模型调用`、`工具接入` 放到一个统一运行时里
- 它的核心目标不是替代单个数据库，而是让应用和 Agent 更容易访问分散在不同系统中的真实数据
- 它支持多种访问方式，官方文档里明确提到可通过 `HTTP`、`Arrow Flight`、`Arrow Flight SQL`、`ODBC`、`JDBC`、`ADBC` 等接口查询
- 它也提供 `OpenAI 兼容 API` 和 `MCP API`，因此既能做数据查询层，也能做 AI/Agent 的后端能力层

更直白一点说，Spice.ai 解决的是下面这类问题：

- 数据在 PostgreSQL、ClickHouse、文件、对象存储等多个地方，应用想统一查询
- 查询能跑，但访问路径很碎，代码里要写很多胶水逻辑
- 想给 Agent 或 LLM 接真实数据、搜索和工具，但不想自己再拼一套后端
- 想做热点数据加速、联邦查询、RAG 或企业内部问答

## 核心能力

结合官方文档，Spice.ai 现在主要覆盖这几类能力：

- 数据集管理：用 `spicepod.yaml` 定义 datasets、models、secrets 等
- 联邦查询：统一查询不同来源的数据集
- 加速：可以对数据集开启 acceleration，加速查询
- 搜索：支持全文/向量搜索能力
- AI 接口：支持 OpenAI 兼容 HTTP API
- Agent 集成：支持 MCP API，把外部工具接入运行时

## 如何使用

### 1. 安装 CLI

官方文档给出的常见安装方式：

```bash
curl https://install.spiceai.org | /bin/bash
```

或者使用 Homebrew：

```bash
brew install spiceai/spiceai/spice
```

### 2. 初始化项目

典型 CLI 流程：

```bash
spice init spice_app
cd spice_app
spice add spiceai/quickstart
spice run
```

这里的关键概念是 `spicepod.yaml`。

一个最小示例大致长这样：

```yaml
version: v1
kind: Spicepod
name: my_spicepod

datasets:
  - from: spice.ai/spiceai/quickstart/datasets/taxi_trips
    name: taxi_trips
    acceleration:
      enabled: true
```

### 3. 定义数据集

Spice.ai 的使用方式，本质上是先在 `spicepod.yaml` 里声明数据源，再启动运行时。

例如可以声明：

- PostgreSQL 表
- ClickHouse 表
- 本地 CSV 文件
- 远程数据集或内置 quickstart 数据集

运行时启动后，这些数据集会被注册成可查询的逻辑表。

### 4. 启动运行时

```bash
spice run
```

启动后，Spice 会：

- 读取 `spicepod.yaml`
- 初始化并注册 datasets
- 对外暴露 SQL / HTTP / Flight / AI API

### 5. 查询或接入应用

运行时启动后，常见使用方式包括：

- 用 SQL 查询数据集
- 用 Arrow Flight / Flight SQL 从程序里访问
- 通过 HTTP API 接 OpenAI 兼容接口
- 通过 `/v1/search` 做搜索
- 通过 MCP API 把能力暴露给 Agent

## 官方文档中常见用例

从官方文档和示例来看，Spice.ai 当前适合的用例主要有：

### 1. 多数据源统一查询

典型场景：

- 一个应用要同时访问 PostgreSQL、ClickHouse、S3、文件数据
- 希望统一用 SQL 查询，而不是每种源单独写一套代码

### 2. 数据加速

对热点或频繁访问数据启用 acceleration，让查询更快。

这类场景适合：

- BI/分析查询
- 高频读场景
- 作为应用附近的数据加速层

### 3. 搜索 / RAG

官方文档里有 `/v1/search` 和向量搜索能力，因此它适合：

- 文档检索
- 混合检索
- 基于企业数据的 RAG 应用

### 4. AI/Agent 后端

因为它提供：

- OpenAI 兼容 API
- MCP API
- 模型配置与工具集成

所以它也适合做：

- 企业内部问答助手
- 数据驱动 Agent
- 带 SQL / memory / tools 的聊天应用

### 当前在做什么

这个用例本质上是在验证：

- Spice 是否能同时接多个异构数据源
- 同一份业务查询是否能在不同数据源上统一执行
- 本地文件、PostgreSQL、ClickHouse 能否通过一个 spicepod 统一暴露

### 当前业务形态

从表名看，这是一套很轻量的“IP -> 城市 / 国家”示例：

- `attacker_ip`
- `city`

- 单表按 `attacker_ip` 查城市
- 表连接按 `city_id` 查国家

也就是说，当前仓库里的 Spice 用例更偏向：

- 异构数据接入验证
- 统一 SQL 查询验证
